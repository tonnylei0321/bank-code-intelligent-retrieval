# 文档翻译指南

## 📋 翻译规范

### 1. 翻译原则

#### 准确性
- 准确理解原文含义，不遗漏、不曲解
- 保持原文的逻辑结构和层次关系
- 保留技术术语的准确性

#### 一致性
- 统一术语翻译（见术语表）
- 保持文档风格一致
- 使用统一的格式和标点符号

#### 可读性
- 使用自然、流畅的中文表达
- 避免生硬的直译
- 保持段落结构清晰

#### 专业性
- 使用专业的技术术语
- 保持文档的专业风格
- 符合中文技术文档规范

### 3. 格式规范

#### 标题翻译
```markdown
# 原文标题
# 中文标题

## 原文子标题
## 中文子标题
```

#### 代码块保留
```markdown
代码块中的代码保持不变，只翻译注释和说明文字

```python
# 中文注释
def function():
    pass
```
```

#### 链接保留
```markdown
[中文链接文本](原始URL)
```

#### 列表翻译
```markdown
- 原文项目1
- 中文项目1

- 原文项目2
- 中文项目2
```

#### 表格翻译
```markdown
| 英文列1 | 英文列2 |
|--------|--------|
| 中文列1 | 中文列2 |
```

### 4. 标点符号规范

#### 中文标点
- 句号：。（不用.）
- 逗号：，（不用,）
- 冒号：：（不用:）
- 分号：；（不用;）
- 问号：？（不用?）
- 感叹号：！（不用!）
- 引号：「」或""（不用""）
- 括号：（）（不用()）

#### 特殊情况
- 英文缩写后保留英文标点：API、HTTP等
- 代码中的标点保持原样
- URL中的标点保持原样

### 5. 数字和单位

#### 数字表示
- 阿拉伯数字：1、2、3等
- 百分比：100%、50%等
- 版本号：v1.0、v2.1等

#### 单位表示
- 时间：小时、分钟、秒
- 大小：字节、KB、MB、GB
- 速度：请求/秒、次/分钟

### 6. 特殊表达

#### 技术概念
- 保留英文原文：API、HTTP、JSON等
- 添加中文解释：API（应用程序接口）
- 首次出现时加注：LoRA（参数高效微调）

#### 用户界面
- 按钮：「按钮名称」
- 菜单：「菜单 > 子菜单」
- 字段：「字段名称」

#### 代码示例
- 保留代码不变
- 翻译说明文字
- 翻译注释

## 📝 翻译流程

### 1. 准备阶段
- [ ] 阅读原文，理解整体内容
- [ ] 查阅术语表，确保术语一致
- [ ] 准备翻译工具（编辑器、词典等）

### 2. 翻译阶段
- [ ] 逐段翻译，保持原文结构
- [ ] 检查术语一致性
- [ ] 验证格式和链接
- [ ] 保留代码块和特殊格式

### 3. 审查阶段
- [ ] 通读全文，检查流畅性
- [ ] 验证术语准确性
- [ ] 检查标点符号
- [ ] 验证链接和格式

### 4. 最终阶段
- [ ] 对比原文，确保完整性
- [ ] 检查拼写和语法
- [ ] 提交审核

## 💡 常见翻译问题

### Q1: 如何翻译"endpoint"？
**A**: 翻译为"端点"。例如："API端点"、"HTTP端点"

### Q2: 如何翻译"request/response"？
**A**: 翻译为"请求/响应"。例如："HTTP请求"、"API响应"

### Q3: 如何翻译"configuration"？
**A**: 翻译为"配置"。例如："系统配置"、"环境配置"

### Q4: 如何翻译"deployment"？
**A**: 翻译为"部署"。例如："系统部署"、"模型部署"

### Q5: 代码中的注释如何翻译？
**A**: 翻译为中文注释，保持代码本身不变。

### Q6: 如何处理缩写词？
**A**: 保留英文缩写，首次出现时加中文解释。例如："API（应用程序接口）"

### Q7: 如何翻译URL和链接？
**A**: 保留URL不变，翻译链接文本。例如："[中文文本](https://example.com)"

### Q8: 如何翻译表格？
**A**: 翻译表头和内容，保持表格结构不变。
